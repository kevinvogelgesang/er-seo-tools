## 0. ¬ß9 Q3 ‚Äî offering DISABLE behavior (**Kevin call ‚Äî BLOCKS F2 merge, not spec/plan work**)

Recommendation (matches roadmap's own "recommended"): disabling an offering **ARCHIVES** offering-exclusive subsection instances (and the fields they own) ‚Äî `archivedAt` + `archiveReason: 'offering'` stamped, data fully preserved, hidden from viewer/admin render; a section instance left with zero live subsections is archived too. **Re-enabling restores the FROZEN snapshot**: archived `'offering'`-reason instances matching the re-enabled offering are un-archived in place exactly as they were (answers/history/local edits intact ‚Äî NO silent refresh to current template; that would bypass ¬ß6's pull confirmation and contradict D8's freeze); template subsections with no instance at all are snapshot-created from the current template. Refresh afterwards is the explicit per-section pull. Operator-archived fields (`archiveReason: 'operator'`) are NEVER auto-restored by re-enable or pull. Archived ack-able sections are excluded from the `pcCompletedAt` completion requirement set (an un-ackable required section would deadlock completion; F3 replaces this gate anyway). Alternative (rejected): hard-delete on disable ‚Äî destroys client answers for a reversible flag flip, and D9's append-only-history philosophy says never.

## 1. Goal

Every viewbook becomes a self-contained **instance tree** snapshotted from the F1 template library at creation (copy-on-create, D8): per-viewbook section + subsection rows carrying their own copy of titles, copy, content, offering tags, order, and **assets** ‚Äî rendering never touches a template or global row again. `Viewbook` gains offering flags (D5); `ViewbookField` becomes subsection-instance-owned; "update to current global version" becomes a **versioned per-section pull-merge** that preserves answers and history. The viewer/admin read model cuts over atomically to instances, THEN the legacy stores (`ViewbookGlobalContent`, `ViewbookContentOverride`), their routes, and the F1b dual-write bridge are retired. All existing viewbooks are test-only and are **wiped in the migration** (D4 ‚Äî no data migration owed).

## 3. Approaches considered

- **A (chosen): evolve `ViewbookSection` into the section instance + new `ViewbookSubsection` table; keep the public payload shape (`data.global`/`data.sectionCopy`) derived from instances.** One row per section keeps state/ack/content together (D6: sections stay the ack unit), reuses `@@unique([viewbookId, sectionKey])` as the durable instance key that already drives anchors/theme/ack/inspector, and the payload-compat loader keeps the 13 section components byte-stable for the parity gate. Wipe (D4) makes the required-column rebuild trivial.
- **B (rejected): separate `ViewbookSectionInstance` table beside `ViewbookSection`.** Two rows per section, every consumer joins state‚Üîcontent, ack/hidden semantics split across tables ‚Äî pure cost, no isolation gain.
- **C (rejected for F2): restructure the public payload to a sections‚Üísubsections shape and rewrite the 13 components now.** That is F3's viewer rebuild; doing it inside the cutover PR destroys the rendered-parity gate and doubles the blast radius.

## 5. Copy-on-create snapshot

`lib/viewbook/instance-snapshot.ts` ‚Äî **ONE pure projection** (F1 fix-#15 pattern): `projectInstanceTree(templateTree, offerings)` ‚Üí `{rows, assetPlan}`: the full nested rows for every ACTIVE (non-archived) template section with ‚â•1 active subsection matching the enabled offerings (D5), containing only matching subsections, each carrying its active `FieldTemplate` rows as `ViewbookField` inputs (`defKey = fieldKey`, `category = subsectionKey`, value null, `createdBy: 'seed'`); **plus `assetPlan: [{sectionKey, subsectionKey, refs: [globalFilename‚Ä¶]}]` captured BEFORE the photo refs are null-stripped from the rows** (Codex fix #8a ‚Äî once phase 1 writes `photo: null`, the stored row no longer knows which source files to copy; the plan is the copy manifest phase 2 consumes). Section `state` initialization preserves today's `createViewbook` semantics exactly: `assessment` starts `'hidden'` for `kind: 'new-build'`, `'active'` otherwise (Codex fix #12 ‚Äî pinned in the ¬ß12 fixture). Used by `createViewbook`, offering-enable (¬ß7), pull's create-missing branch (¬ß6), and the tests.

`createViewbook(clientId, kind, offerings, createdBy)` becomes two phases:
1. **One nested `prisma.viewbook.create`** (array-form txn as today): viewbook row (+ offering flags) + section instances + subsection instances + fields + `DEFAULT_MILESTONES`. Roster photo refs in subsection content are written as `photo: null` in this phase (never a `'global'` filename ‚Äî a global name in instance content would either break on template replace or leak template-current files; `TeamMember.photo` is nullable so the render degrades to photoless, honestly).
2. **Best-effort asset snapshot** (`snapshotInstanceAssets`, ¬ß8): driven by the phase-1 `assetPlan` (not the stored rows; entries map each copied file to its roster member ‚Äî `{sectionKey, subsectionKey, refs: [{memberName, filename}]}` ‚Äî so the rewrite lands on the right member), copy each referenced global file into the new viewbook's scope, then a per-subsection fenced txn rewrites `contentJson` with the new filenames + bumps the owning section's aggregate `version` + **the viewbook's scoped syncVersion** (the rewrite changes rendered content ‚Äî an open tab must refetch; fence on subsection `version`; fence loss ‚Üí delete the new files). A phase-2 failure logs and leaves photos null ‚Äî the **always-available "Refresh from template" pull repairs it** (¬ß6; Codex fix #9 ‚Äî repair must not depend on the template being NEWER, because a failed phase 2 leaves `templateVersion` already equal to current); the viewbook is never broken, only photoless. (Scope = the viewbook id, which does not exist before phase 1 ‚Äî full single-txn atomicity is impossible without interactive transactions, which are banned.)

`syncCatalogQuestions` + `POST /api/viewbooks/[id]/sync-questions` are **retired** ‚Äî pull (¬ß6) is the backfill path.

## 6. Pull ‚Äî "update to current template version" (versioned MERGE, per section)

`POST /api/viewbooks/[id]/sections/[sectionKey]/pull` `{version}` ‚Üí `pullSectionFromTemplate()` in the new `lib/viewbook/instance-service.ts`. Preconditions: instance exists; `sectionTemplateId` non-null (else 409 `template_missing` ‚Äî SetNull'd/never-linked instances can't pull); template section not archived (409 `template_archived`). **Pull is deliberately available at EQUAL versions** ("Refresh from template" ‚Äî the ¬ß5 photoless-repair path and a general re-sync; the update badge signals a newer template, it does not gate the action).

**Merge semantics (the roadmap fix-#5 contract, pinned):**
- **Section scalars:** `title`/`rendererType`/`copyJson`/`contentJson` ‚Üê template; `templateVersion` ‚Üê current `SectionTemplate.version`; instance `version`++. `state`/`introNote`/`narrative`/`acknowledgedAt`/`doneAt` are per-viewbook state ‚Äî NEVER touched by pull.
- **Subsections** diffed by `subsectionKey` against the template's ACTIVE subsections filtered to the viewbook's enabled offerings:
  - *Both sides:* overwrite `title`/`copyJson`/`contentJson`/offering booleans; `version`++; `archivedAt`/`archiveReason` cleared (the match set is already offering-filtered, so an offering-archived subsection outside the filter never appears here).
  - *Template-only:* snapshot-create the instance (+ its fields + assets).
  - *Instance-only (template counterpart archived or gone):* stamp `archivedAt` ‚Äî **archive, never delete** (answers + history preserved).
  - *Empty after filtering:* if the template section has ZERO active subsections matching the viewbook's offerings, pull archives every instance subsection AND the section itself (`archiveReason: 'pull'`) ‚Äî mirrors ¬ß7's zero-live-subsections rule; never a 500, never a delete.
  - *Ownership predicates:* every field archive/update statement is additionally predicated on the field's CURRENT owning subsection belonging to THIS section (`WHERE ‚Ä¶ AND "subsectionId" IN (SELECT id FROM "ViewbookSubsection" WHERE "sectionId" = ?)`) ‚Äî a stale pull of section A must not archive a field a concurrent pull already moved to section B.
- **Fields** matched by `defKey == fieldKey` **viewbook-globally** (the `@@unique([viewbookId, defKey])` join): existing ‚Üí update `label`/`sortOrder`, re-parent `subsectionId` + `category` to the current owning subsection, clear `archivedAt` **unless `archiveReason: 'operator'`** (operator archives are never auto-restored); **`value`/`version`/amendment history are NEVER touched**; missing ‚Üí create (`createdBy: 'pull'`); instance fields under this section's subsections whose defKey has no active `FieldTemplate` in this section's template ‚Üí archive (`archiveReason: 'pull'`; a field that MOVED to another template section is restored + re-parented when THAT section is pulled ‚Äî defKey matching is viewbook-global, so no duplicate row can exist). `fieldType` is immutable template-side (F1 has no PATCH surface) ‚Äî pull never changes it; custom fields (`defKey null`) are untouched.
- **Local-edit policy (pinned): pull OVERWRITES instance content/copy/titles wholesale, behind a UI confirmation** ("replaces this section's content with the current template; answers, history, and completion are kept"). Selective merge is rejected ‚Äî it is an F5b-scale diff UI for marginal value, and D8's model is freeze + explicit pull, not three-way merge.
- **Mechanics:** asset pre-copy (new files into viewbook scope) ‚Üí ONE array-form txn (throwing conditional-update guards fenced on the request's expected section `version` ‚Äî which is the AGGREGATE content-tree fence per ¬ß4, so any concurrent subsection/field content mutation invalidates the pull with a 409 instead of being silently overwritten (Codex fix #2); P2002 on subsection/field creates rolls back whole). **In-txn ID addressing (Codex fix #3):** a pull that creates a subsection AND re-parents an existing field into it cannot know the new autoincrement id while building the array ‚Äî those statements are pinned to **raw SQL with durable-key subselects** (`INSERT ‚Ä¶ SELECT`/`UPDATE ‚Ä¶ SET "subsectionId" = (SELECT id FROM "ViewbookSubsection" WHERE "viewbookId" = ? AND "sectionId" = ? AND "subsectionKey" = ?)`, the C14/F1b raw-statement precedent; nested Prisma creates would P2002 on the existing defKey instead of merging). Post-commit: best-effort delete of replaced instance-scope files **only when a filename is absent from the complete post-commit asset-reference union of the whole viewbook** (¬ß8; Codex fix #8c ‚Äî another subsection may still reference it); txn failure deletes the NEW files. Scoped syncVersion bump inside the txn. Response: `{summary: {subsectionsAdded/Updated/Archived, fieldsAdded/Updated/Archived}}` + the refreshed section tree.
- **"Update available" indicator:** `getViewbookAdmin` serves each instance's `templateVersion` alongside the current template `version`; ContentTab badges sections where they differ (cheap read, no polling).

## 7. Offering flags ‚Äî enable/disable after creation

`PATCH /api/viewbooks/[id]/offerings` `{offeringWebsite, offeringVa, offeringPpc, expected: {‚Ä¶current flags‚Ä¶}}` (operator; ‚â•1 true or 400 `invalid_offerings`). Creation UI gains the three checkboxes (website pre-checked).

- **One fenced operation (Codex fix #4):** the flag update, every section/subsection archive/restore, every field archive/restore, and the scoped syncVersion bump commit in ONE array-form txn whose statements are all predicated on the viewbook still holding the `expected` pre-state flags (mismatch ‚Üí 0-count guard throws ‚Üí 409 `offerings_conflict`) ‚Äî a crash or concurrent PATCH can never leave flags and the instance tree disagreeing. **Statement order is load-bearing:** all companion mutations (predicated on the PRE-state flags) run FIRST; the throwing guarded flag update runs LAST ‚Äî if the flag flip ran first, every later pre-state-predicated statement would silently no-op instead of rolling back.
- **Offering availability gate (Codex fix #6):** enabling an offering with ZERO active matching template subsections (today: anything but website) ‚Üí 409 `offering_unavailable` ‚Äî never a silently-empty viewbook. Creation applies the same rule (the create form disables unavailable checkboxes; availability derived from the template tree server-side).
- **Enable:** for each active template section with ‚â•1 active subsection matching the NEW offering set: no section instance ‚Üí snapshot-create from the CURRENT template (mixed-vintage trees are legal; pull normalizes); instance exists ‚Üí archived `'offering'`-reason subsections matching the re-enabled offering are **un-archived AS FROZEN** (Codex fix #5 ‚Äî no content refresh; ¬ß6's pull confirmation is the only refresh path), their owned fields un-archived EXCEPT `archiveReason: 'operator'`; template subsections with no instance at all are snapshot-created; an archived section instance that regains a live subsection is un-archived.
- **Disable:** every live subsection instance whose SNAPSHOT offering tags no longer intersect the enabled set ‚Üí `archivedAt` + `archiveReason: 'offering'` (+ its non-archived fields, same reason); a section left with zero live subsections ‚Üí archived, same reason. **Data preserved** per ¬ß0's recommendation.
- **`archivedAt` joins every behavioral fence, not just readers (Codex fix #7):** today's write fences test only `state <> 'hidden'` (`acknowledgeSection`, `applyAnswerEdit`/amendments, `setNotifyEmails`, `addTeamMember`, `insertClientFeedback`/`insertClientMaterial` where section-scoped, ack reset, collapse) ‚Äî each gains `archivedAt IS NULL` on the section (and subsection for field writes), so an archived tree is inert for clients AND operators until restored. The OPERATOR surfaces fence too: `setSectionState`/`updateSectionText` reject archived sections, and `operator-data.ts` excludes archived sections/subsections from the inspector payload. Each route keeps its EXISTING hidden-section error shape for the archived case (same code, same status) ‚Äî archived behaves exactly like hidden at the API boundary. Completion counting: archived ack-able sections drop out of the `pcCompletedAt` requirement set (¬ß0). Renderer/admin/diagnostics filter `archivedAt: null` everywhere (¬ß8/¬ß9).
- Asset copies for newly-created subsections follow ¬ß5 phase-2 (degrade-to-null + equal-version pull repairs).

## 8. Asset snapshot layer (roadmap fix #5c: "copying a filename is NOT a snapshot")

- **`lib/viewbook/instance-asset-refs.ts` ‚Äî pure, THE single home** of `extractInstanceAssetRefs(rendererType, contentJson) ‚Üí string[]` (today: welcome-renderer roster photos; future renderers add here). Consumed by ALL FOUR seams so they cannot drift (the admin-polish lesson, now structural): (1) snapshot/pull asset copy, (2) the serving-route allowlist, (3) the retention union, (4) the delete-snapshot.
- **Copy:** `snapshotInstanceAssets` reads each source file (`readViewbookAsset('global', f)` at create/enable; the template's global-scope roster files remain the source of truth for templates), saves a NEW uuid file into `String(viewbookId)` scope, rewrites the subsection `contentJson`. Missing/corrupt source ‚Üí that ref becomes null + `logError` (honest degrade). Instances therefore reference ONLY their own scope ‚Äî the template photo replace flow (which deletes the old global file) can no longer break any viewbook, which is the freeze F1 ¬ß9 promised.
- **Serving allowlist** (`/api/viewbook/[token]/assets/[filename]`): add lookup (5) ‚Äî filenames extracted from the token's own live subsection instances, viewbook scope. **Remove the global team-roster branch** ‚Äî post-cutover no public surface references global roster files (verified: the template admin renders no photo preview), and keeping it would serve template-current files to any token holder (curation drift). Same-404 discipline unchanged.
- **Retention union** (`pruneOrphanedViewbookAssetFiles`): add subsection-instance refs as the 5th lookup ‚Äî computed over **ALL subsections including archived ones** (Codex fix #8b: serving is live-only, but the 24 h orphan sweep must not destroy files an archived subsection needs for a disable‚Üíenable restore). Fault isolation (any-throw-aborts-scope) preserved. **Delete-snapshot:** `deleteViewbook` + `collectClientViewbookAssetSnapshot` add instance refs (archived included) to the filename union. **Replacement deletes** (pull/¬ß6) remove an old file only when it is absent from the complete post-commit reference union of the viewbook (Codex fix #8c).

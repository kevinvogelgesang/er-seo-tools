## 0. §9 Q3 — offering DISABLE behavior (**Kevin call — BLOCKS F2 merge, not spec/plan work**)

Recommendation (matches roadmap's own "recommended"): disabling an offering **ARCHIVES** offering-exclusive subsection instances (and the fields they own) — `archivedAt` + `archiveReason: 'offering'` stamped, data fully preserved, hidden from viewer/admin render; a section instance left with zero live subsections is archived too. **Re-enabling restores the FROZEN snapshot**: archived `'offering'`-reason instances matching the re-enabled offering are un-archived in place exactly as they were (answers/history/local edits intact — NO silent refresh to current template; that would bypass §6's pull confirmation and contradict D8's freeze); template subsections with no instance at all are snapshot-created from the current template. Refresh afterwards is the explicit per-section pull. Operator-archived fields (`archiveReason: 'operator'`) are NEVER auto-restored by re-enable or pull. Archived ack-able sections are excluded from the `pcCompletedAt` completion requirement set (an un-ackable required section would deadlock completion; F3 replaces this gate anyway). Alternative (rejected): hard-delete on disable — destroys client answers for a reversible flag flip, and D9's append-only-history philosophy says never.

## 1. Goal

Every viewbook becomes a self-contained **instance tree** snapshotted from the F1 template library at creation (copy-on-create, D8): per-viewbook section + subsection rows carrying their own copy of titles, copy, content, offering tags, order, and **assets** — rendering never touches a template or global row again. `Viewbook` gains offering flags (D5); `ViewbookField` becomes subsection-instance-owned; "update to current global version" becomes a **versioned per-section pull-merge** that preserves answers and history. The viewer/admin read model cuts over atomically to instances, THEN the legacy stores (`ViewbookGlobalContent`, `ViewbookContentOverride`), their routes, and the F1b dual-write bridge are retired. All existing viewbooks are test-only and are **wiped in the migration** (D4 — no data migration owed).

## 3. Approaches considered

- **A (chosen): evolve `ViewbookSection` into the section instance + new `ViewbookSubsection` table; keep the public payload shape (`data.global`/`data.sectionCopy`) derived from instances.** One row per section keeps state/ack/content together (D6: sections stay the ack unit), reuses `@@unique([viewbookId, sectionKey])` as the durable instance key that already drives anchors/theme/ack/inspector, and the payload-compat loader keeps the 13 section components byte-stable for the parity gate. Wipe (D4) makes the required-column rebuild trivial.
- **B (rejected): separate `ViewbookSectionInstance` table beside `ViewbookSection`.** Two rows per section, every consumer joins state↔content, ack/hidden semantics split across tables — pure cost, no isolation gain.
- **C (rejected for F2): restructure the public payload to a sections→subsections shape and rewrite the 13 components now.** That is F3's viewer rebuild; doing it inside the cutover PR destroys the rendered-parity gate and doubles the blast radius.

## 5. Copy-on-create snapshot

`lib/viewbook/instance-snapshot.ts` — **ONE pure projection** (F1 fix-#15 pattern): `projectInstanceTree(templateTree, offerings)` → `{rows, assetPlan}`: the full nested rows for every ACTIVE (non-archived) template section with ≥1 active subsection matching the enabled offerings (D5), containing only matching subsections, each carrying its active `FieldTemplate` rows as `ViewbookField` inputs (`defKey = fieldKey`, `category = subsectionKey`, value null, `createdBy: 'seed'`); **plus `assetPlan: [{sectionKey, subsectionKey, refs: [globalFilename…]}]` captured BEFORE the photo refs are null-stripped from the rows** (Codex fix #8a — once phase 1 writes `photo: null`, the stored row no longer knows which source files to copy; the plan is the copy manifest phase 2 consumes). Section `state` initialization preserves today's `createViewbook` semantics exactly: `assessment` starts `'hidden'` for `kind: 'new-build'`, `'active'` otherwise (Codex fix #12 — pinned in the §12 fixture). Used by `createViewbook`, offering-enable (§7), pull's create-missing branch (§6), and the tests.

`createViewbook(clientId, kind, offerings, createdBy)` becomes two phases:
1. **One nested `prisma.viewbook.create`** (array-form txn as today): viewbook row (+ offering flags) + section instances + subsection instances + fields + `DEFAULT_MILESTONES`. Roster photo refs in subsection content are written as `photo: null` in this phase (never a `'global'` filename — a global name in instance content would either break on template replace or leak template-current files; `TeamMember.photo` is nullable so the render degrades to photoless, honestly).
2. **Best-effort asset snapshot** (`snapshotInstanceAssets`, §8): driven by the phase-1 `assetPlan` (not the stored rows), copy each referenced global file into the new viewbook's scope, then a per-subsection fenced txn rewrites `contentJson` with the new filenames + bumps the owning section's aggregate `version` (fence on subsection `version`; fence loss → delete the new files). A phase-2 failure logs and leaves photos null — the **always-available "Refresh from template" pull repairs it** (§6; Codex fix #9 — repair must not depend on the template being NEWER, because a failed phase 2 leaves `templateVersion` already equal to current); the viewbook is never broken, only photoless. (Scope = the viewbook id, which does not exist before phase 1 — full single-txn atomicity is impossible without interactive transactions, which are banned.)

`syncCatalogQuestions` + `POST /api/viewbooks/[id]/sync-questions` are **retired** — pull (§6) is the backfill path.

## 6. Pull — "update to current template version" (versioned MERGE, per section)

`POST /api/viewbooks/[id]/sections/[sectionKey]/pull` `{version}` → `pullSectionFromTemplate()` in the new `lib/viewbook/instance-service.ts`. Preconditions: instance exists; `sectionTemplateId` non-null (else 409 `template_missing` — SetNull'd/never-linked instances can't pull); template section not archived (409 `template_archived`). **Pull is deliberately available at EQUAL versions** ("Refresh from template" — the §5 photoless-repair path and a general re-sync; the update badge signals a newer template, it does not gate the action).

**Merge semantics (the roadmap fix-#5 contract, pinned):**
- **Section scalars:** `title`/`rendererType`/`copyJson`/`contentJson` ← template; `templateVersion` ← current `SectionTemplate.version`; instance `version`++. `state`/`introNote`/`narrative`/`acknowledgedAt`/`doneAt` are per-viewbook state — NEVER touched by pull.
- **Subsections** diffed by `subsectionKey` against the template's ACTIVE subsections filtered to the viewbook's enabled offerings:
  - *Both sides:* overwrite `title`/`copyJson`/`contentJson`/offering booleans; `version`++; `archivedAt`/`archiveReason` cleared (the match set is already offering-filtered, so an offering-archived subsection outside the filter never appears here).
  - *Template-only:* snapshot-create the instance (+ its fields + assets).
  - *Instance-only (template counterpart archived or gone):* stamp `archivedAt` — **archive, never delete** (answers + history preserved).
- **Fields** matched by `defKey == fieldKey` **viewbook-globally** (the `@@unique([viewbookId, defKey])` join): existing → update `label`/`sortOrder`, re-parent `subsectionId` + `category` to the current owning subsection, clear `archivedAt` **unless `archiveReason: 'operator'`** (operator archives are never auto-restored); **`value`/`version`/amendment history are NEVER touched**; missing → create (`createdBy: 'pull'`); instance fields under this section's subsections whose defKey has no active `FieldTemplate` in this section's template → archive (`archiveReason: 'pull'`; a field that MOVED to another template section is restored + re-parented when THAT section is pulled — defKey matching is viewbook-global, so no duplicate row can exist). `fieldType` is immutable template-side (F1 has no PATCH surface) — pull never changes it; custom fields (`defKey null`) are untouched.
- **Local-edit policy (pinned): pull OVERWRITES instance content/copy/titles wholesale, behind a UI confirmation** ("replaces this section's content with the current template; answers, history, and completion are kept"). Selective merge is rejected — it is an F5b-scale diff UI for marginal value, and D8's model is freeze + explicit pull, not three-way merge.
- **Mechanics:** asset pre-copy (new files into viewbook scope) → ONE array-form txn (throwing conditional-update guards fenced on the request's expected section `version` — which is the AGGREGATE content-tree fence per §4, so any concurrent subsection/field content mutation invalidates the pull with a 409 instead of being silently overwritten (Codex fix #2); P2002 on subsection/field creates rolls back whole). **In-txn ID addressing (Codex fix #3):** a pull that creates a subsection AND re-parents an existing field into it cannot know the new autoincrement id while building the array — those statements are pinned to **raw SQL with durable-key subselects** (`INSERT … SELECT`/`UPDATE … SET "subsectionId" = (SELECT id FROM "ViewbookSubsection" WHERE "viewbookId" = ? AND "sectionId" = ? AND "subsectionKey" = ?)`, the C14/F1b raw-statement precedent; nested Prisma creates would P2002 on the existing defKey instead of merging). Post-commit: best-effort delete of replaced instance-scope files **only when a filename is absent from the complete post-commit asset-reference union of the whole viewbook** (§8; Codex fix #8c — another subsection may still reference it); txn failure deletes the NEW files. Scoped syncVersion bump inside the txn. Response: `{summary: {subsectionsAdded/Updated/Archived, fieldsAdded/Updated/Archived}}` + the refreshed section tree.
- **"Update available" indicator:** `getViewbookAdmin` serves each instance's `templateVersion` alongside the current template `version`; ContentTab badges sections where they differ (cheap read, no polling).

## 7. Offering flags — enable/disable after creation

`PATCH /api/viewbooks/[id]/offerings` `{offeringWebsite, offeringVa, offeringPpc, expected: {…current flags…}}` (operator; ≥1 true or 400 `invalid_offerings`). Creation UI gains the three checkboxes (website pre-checked).

- **One fenced operation (Codex fix #4):** the flag update, every section/subsection archive/restore, every field archive/restore, and the scoped syncVersion bump commit in ONE array-form txn whose statements are all predicated on the viewbook still holding the `expected` pre-state flags (mismatch → 0-count guard throws → 409 `offerings_conflict`) — a crash or concurrent PATCH can never leave flags and the instance tree disagreeing.
- **Offering availability gate (Codex fix #6):** enabling an offering with ZERO active matching template subsections (today: anything but website) → 409 `offering_unavailable` — never a silently-empty viewbook. Creation applies the same rule (the create form disables unavailable checkboxes; availability derived from the template tree server-side).
- **Enable:** for each active template section with ≥1 active subsection matching the NEW offering set: no section instance → snapshot-create from the CURRENT template (mixed-vintage trees are legal; pull normalizes); instance exists → archived `'offering'`-reason subsections matching the re-enabled offering are **un-archived AS FROZEN** (Codex fix #5 — no content refresh; §6's pull confirmation is the only refresh path), their owned fields un-archived EXCEPT `archiveReason: 'operator'`; template subsections with no instance at all are snapshot-created; an archived section instance that regains a live subsection is un-archived.
- **Disable:** every live subsection instance whose SNAPSHOT offering tags no longer intersect the enabled set → `archivedAt` + `archiveReason: 'offering'` (+ its non-archived fields, same reason); a section left with zero live subsections → archived, same reason. **Data preserved** per §0's recommendation.
- **`archivedAt` joins every behavioral fence, not just readers (Codex fix #7):** today's write fences test only `state <> 'hidden'` (`acknowledgeSection`, `applyAnswerEdit`/amendments, `setNotifyEmails`, `addTeamMember`, `insertClientFeedback`/`insertClientMaterial` where section-scoped, ack reset, collapse) — each gains `archivedAt IS NULL` on the section (and subsection for field writes), so an archived tree is inert for clients AND operators until restored. Completion counting: archived ack-able sections drop out of the `pcCompletedAt` requirement set (§0). Renderer/admin/diagnostics filter `archivedAt: null` everywhere (§8/§9).
- Asset copies for newly-created subsections follow §5 phase-2 (degrade-to-null + equal-version pull repairs).

## 8. Asset snapshot layer (roadmap fix #5c: "copying a filename is NOT a snapshot")

- **`lib/viewbook/instance-asset-refs.ts` — pure, THE single home** of `extractInstanceAssetRefs(rendererType, contentJson) → string[]` (today: welcome-renderer roster photos; future renderers add here). Consumed by ALL FOUR seams so they cannot drift (the admin-polish lesson, now structural): (1) snapshot/pull asset copy, (2) the serving-route allowlist, (3) the retention union, (4) the delete-snapshot.
- **Copy:** `snapshotInstanceAssets` reads each source file (`readViewbookAsset('global', f)` at create/enable; the template's global-scope roster files remain the source of truth for templates), saves a NEW uuid file into `String(viewbookId)` scope, rewrites the subsection `contentJson`. Missing/corrupt source → that ref becomes null + `logError` (honest degrade). Instances therefore reference ONLY their own scope — the template photo replace flow (which deletes the old global file) can no longer break any viewbook, which is the freeze F1 §9 promised.
- **Serving allowlist** (`/api/viewbook/[token]/assets/[filename]`): add lookup (5) — filenames extracted from the token's own live subsection instances, viewbook scope. **Remove the global team-roster branch** — post-cutover no public surface references global roster files (verified: the template admin renders no photo preview), and keeping it would serve template-current files to any token holder (curation drift). Same-404 discipline unchanged.
- **Retention union** (`pruneOrphanedViewbookAssetFiles`): add subsection-instance refs as the 5th lookup — computed over **ALL subsections including archived ones** (Codex fix #8b: serving is live-only, but the 24 h orphan sweep must not destroy files an archived subsection needs for a disable→enable restore). Fault isolation (any-throw-aborts-scope) preserved. **Delete-snapshot:** `deleteViewbook` + `collectClientViewbookAssetSnapshot` add instance refs (archived included) to the filename union. **Replacement deletes** (pull/§6) remove an old file only when it is absent from the complete post-commit reference union of the viewbook (Codex fix #8c). (Known pre-existing gap — assessment images are snapshotted by the route, not `deleteViewbook` — is NOT F2's to fix; noted so the plan doesn't "helpfully" refactor it.)

## 9. Read-model cutover (atomic within the PR)

- **`loadViewbookPublicData`:** the sections query selects the instance columns (+ live subsections); **`PublicSection` gains `title` + `rendererType`** (Codex fix #10 — the type must carry what the switch and `SectionShell` now consume; the §12 fixture normalizes these deliberate additions). `data.sectionCopy` becomes an honest PARTIAL map keyed by the viewbook's OWN sectionKeys (no longer an unconditional 13-key `Record`), built from instance `copyJson` via the existing `toLegacySectionCopy` translator; **corrupt/unparseable instance `copyJson` falls back to `SECTION_COPY[key]` for the 13 known keys (empty copy otherwise) + `logError`** — never a blank ⓘ from one bad row. `data.global` (same `PublicGlobalContent` shape — team/blocks/pcIntro) is assembled from the welcome/strategy/milestones/pc-intro instance subsection content through the `BRIDGED_CONTENT` mapping, which is RETAINED and renamed (`INSTANCE_CONTENT_SLOTS`, moved next to the parsers in `template-content.ts`) — it stops being a bridge map and becomes the instance→payload projection. `data.overrides` is DELETED from the payload; StrategySection/MilestonesSection drop their two override-merge lines (no rendered change post-wipe: there are no override rows). - **`getViewbookAdmin`:** drop the `contentOverrides` include + legacy sectionCopy resolve; serve the instance tree (sections + live subsections, decoded content, `templateVersion` vs current template version). `operator-data.ts` is untouched (reads no content).

## 10. syncVersion policy (redefined — supersedes F1b plan D-e)

Rendering now depends ONLY on instance rows, so: **instance mutations (content/copy/title edits, pull, offerings change, custom-field ops) bump THEIR viewbook's syncVersion, scoped; template mutations bump NOTHING** (template edits reach clients only through pull/enable).

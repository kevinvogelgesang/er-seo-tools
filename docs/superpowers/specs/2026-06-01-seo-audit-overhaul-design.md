# SEO Audit Tool Overhaul — Design Spec

**Date:** 2026-06-01
**Status:** Active (spec)
**Author:** Claude (brainstorming) + 8-agent consensus + Codex review + Kevin's decisions
**Related:** `docs/superpowers/todos/2026-06-01-seo-audit-consensus.md` (consensus report), `docs/superpowers/nyi/specs/FUTURE-keyword-research-tool-design.md` (superseded by Phase 6 here)

---

## 1. Problem

The `/seo-parser` ("SEO Audit" / "Technical Audit") tool ingests Screaming Frog CSV exports, runs ~40 parsers, aggregates into one `AggregatedResult`, computes a health score, stores the result as a JSON-string blob on a `Session` row, and renders a results page. It works end-to-end, but Kevin uses **only** the upload + "copy JSON" steps — he pastes the JSON into a Claude Desktop project to produce the actual roadmap, and separately feeds SEMRush exports into another Claude Desktop project for keyword research. The in-app results page adds little over the raw JSON, so it goes unused.

**Core diagnosis (Codex, adopted over the consensus framing):** the tool is not broken — it is *lopsided*. The parser registry, route flow, and aggregation are sound. The missing piece is the **decision loop**: the product stops at "show aggregate widgets and export JSON," but Kevin's real work *starts* there. The JSON-blob data model is a real future constraint, but it is **not** the root problem and does not block the highest-value fixes.

## 2. Goal

Turn the tool into something Kevin actually uses, that produces:
- **(a)** a prioritized roadmap of real technical-SEO fixes, generated by a Claude skill and rendered back *in the app*;
- **(b)** Teamwork tasks from that roadmap;
- **(c)** keyword-research decision-making (separate, later phase).

Inputs: Screaming Frog + SEMRush now; Node-script live-URL checks and DataForSEO later (typed hooks designed now, no spend).

## 3. Fixed decisions (constraints)

No in-app Anthropic billing. |

## 4. Non-goals

- No rewrite of the parser registry (`BaseParser` + `PARSERS[]` 3-stage detection is kept — it's the strongest part).
- No standalone "refactor `AggregatorService`" project — decompose it **opportunistically** as features force each area open.
- No in-app LLM calls, no Anthropic billing.
- No multi-tenant auth, no real DataForSEO/Node-check spend this cycle.
- No immediate full normalization/backfill of historical sessions.

## 5. Architecture overview

The redesign is **phased**, each phase shippable and useful on its own. The spine is the **decision loop**, not the data model.

```
Upload (SF + optional SEMRush)
   │
   ▼
Parse → AggregatorService → AggregatedResult (blob, unchanged in P1)
   │                         + typed PageIndexEntry[] shape (P1 in-memory/payload; P3 persisted)
   ▼
Results page
   ├─ Severity summary (replaces health score)            [P1]
   ├─ "Suggested priorities" (priority.service, labeled)  [P1]
   ├─ Issue tabs / dupes / keyword signals (existing)
   ├─ Primary action: "Copy for Claude" (trimmed payload) [P1]
   └─ "Generate Roadmap" → mint token ───────────────┐    [P2]
                                                      ▼
                              Claude Desktop/Code skill (seo-audit-roadmap)
                                  GET payload → writes roadmap → PATCH back
                                                      │
                                                      ▼
                              SeoRoadmap rendered in app                     [P2]
                                  └─ "Push to Teamwork" (human-confirmed)    [P4]
```

## 5a. URL model (`UrlRegistry`) — the compaction contract

Path-only-under-one-host compaction is **insufficient** because issues reference URLs not in `internal_all.csv`: external broken links, redirect targets, image/CSS/JS/PDF resources, sitemap URLs, and hreflang alternates on other hosts/subdomains. The compaction layer is therefore a **registry**, not a bare path list:

```ts
UrlRegistry {
  sessionOrigin: { scheme: string; host: string }     // canonical crawl origin
  hosts: string[]                                      // all hosts seen (indexable by hostId)
  urls: Array<{
    id: number                                         // UrlRef target
    kind: 'page' | 'resource' | 'external' | 'redirect-target' | 'sitemap' | 'hreflang'
    hostId: number                                     // index into hosts[]
    path: string                                       // path (host omitted only for sessionOrigin host)
    query?: string                                     // preserved by default (see normalization below)
    originalUrl?: string                               // fallback for anything that can't be safely reconstructed
  }>
}
type UrlRef = number   // index into UrlRegistry.urls
```

- **Issues/recommendations carry `affectedUrlRefs: UrlRef[]`** as the complete-set contract. The legacy `issue.urls: string[]` is kept **only** for display/sample/back-compat.
- **`affectedSetHash`** (used for Teamwork idempotency) hashes the **compact canonical refs**, never display URLs — so host rehydration or normalization tweaks don't spawn duplicate tasks.
- **Normalization rules (conservative, D6):** preserve query strings by default; **drop fragments**; **lowercase host only** (never the path); **do not** strip trailing slashes unless Kevin approves URL-equivalence rules; strip UTM params **only if** both original and canonical forms stay traceable (keep `originalUrl`).
- **Subdomains:** `www.example.edu`, `apply.example.edu`, `catalog.example.edu` are treated as **distinct hosts** in `hosts[]` (do NOT conflate under one session host) unless Kevin decides otherwise — see §11.
- **Payload size:** the export carries a per-issue **summary (counts) + complete `affectedUrlRefs`**; the skill instructions document how to handle long ref lists. Resource/external URLs always resolve through `hostId`/`originalUrl`, never path-only under the session host.

## 6. Phased design

### Phase 1 — Immediate product repair + compaction layer (no schema change)
Make the existing tool worth opening, and build the complete-but-compact payload the P2 roadmap skill depends on. **No DB migration** — all new structures live in the aggregation layer + export payload (persisted in P3).

1. **Build the `UrlRegistry` + `PageIndexEntry[]` in aggregation (no schema).** Generalize `InternalParser.parsePerUrlForPillar()` into a `PageIndexEntry` builder; build the `UrlRegistry` (§5a) from all parser outputs. **Compute complete issue→URL membership**: either parsers emit complete `affectedUrlRefs` into the registry, or a **post-aggregation issue-membership builder** derives `affectedUrlRefs` from the page index + parser outputs. This removes the dependency on today's capped (20/50) `issue.urls` arrays for completeness.
2. **`buildTechnicalAuditExport()` carries complete compact refs.** The export embeds the `UrlRegistry` + per-issue `affectedUrlRefs` (complete) + summary counts. This is the contract P2 consumes — so P2 builds roadmaps on complete evidence, not samples.
3. **Primary copy path uses the trimmed payload.** `CopyToClipboard` currently copies the full `AggregatedResult`. Change the prominent action to copy `buildTechnicalAuditExport(result)` **plus a short invocation instruction block** (what skill/prompt to run). Demote raw-JSON copy and the Summary/Markdown export buttons into a collapsed "Export ▾" menu (raw JSON kept for debugging only).
4. **Fix `optimization_gaps` title/H1 join** (aggregator.service.ts:901–905). The data already exists via `parsePerUrlForPillar()` — now via the new `PageIndexEntry` builder. Populate `title`/`h1` from the page index inside `computeKeywordSignals()`.
5. **Wire `priority.service` into a "Suggested priorities" block.** Render `getPrioritySummary().top_priorities` + `quick_wins` at the top of results. **Label it "Suggested — heuristic ranking of issue types,"** not a score, not work items, not a Teamwork feed. (Codex: weights are static, count-scaling is blunt, quality depends on dedup — useful triage hint, not authoritative.)
6. **Drop the health score cleanly (D8).** Stop computing/writing `metadata.health_score` on new sessions (mark deprecated); hide it in the UI; exclude it from `buildTechnicalAuditExport()` (keep only in raw-debug export). Show a severity summary (critical/warnings/notices) in the hero instead. UI must tolerate old sessions that still carry a score.
7. **URL-list disclosure (resolves the D6/preview tension).** Internal compact refs in the payload are **complete**; the **UI** may render a capped preview labeled "showing N of M". Inventory which parsers cap and at what limit so the membership builder (step 1) backfills the complete set.
8. **Upload data-completeness checklist.** On the upload page, list the SF (and optional SEMRush) exports the parsers consume; after parse, show "Data completeness: N/<available> parsers matched" with which high-value inputs are missing (e.g. "No PageSpeed, no Search Console").

**Exit criteria:** Kevin uploads, sees suggested priorities + severity at a glance, one-click copies a *complete, compact* payload for Claude. The compaction/membership shapes are proven in the payload before any table exists.

### Phase 2 — Close the loop (Technical-Audit roadmap handoff)
Replicate the proven pillar-analysis pattern for the technical audit.

- **New model `SeoRoadmap`** (mirrors `PillarAnalysis`): `sessionId` FK, `status` (`pending|processing|complete|error`), `aiRoadmapMarkdown TEXT?`, `structured JSON?`, `tokenMintedAt`, `narrativeUpdatedAt`. **Status lifecycle** (Claude Desktop is external, so "processing" ≠ background work): `pending` = row created, no token yet; `processing` = token minted / prompt copied (awaiting skill); `complete` = PATCH write-back received; `error` = explicit failure or token expired without write-back. Last-attempt timestamp surfaced in UI.
- **Single scoped token authorizes BOTH** `GET /api/seo-roadmap/[id]/payload` **and** `PATCH /api/seo-roadmap/[id]` — bound to one roadmap + session, short TTL (copy pillar's JWT mint + Bearer auth pattern; do **not** abstract a shared auth layer mid-build — copy it).
- **GET payload endpoint** returns `buildTechnicalAuditExport(result)` (now carrying complete compact refs from P1) for the authenticated skill.
- **PATCH write-back endpoint** stores the returned roadmap markdown + structured roadmap; flips status to `complete`.
- **`seo-audit-roadmap` Claude skill** (new, parallel to `pillar-analysis-narrative`): GET payload → produce structured roadmap (issue → severity → effort → fix guidance → affected URLs) + readable markdown → PATCH back.
- **Results page**: "Generate Roadmap" button (mints token + copies invocation, like pillar's CopyClaudePromptButton) → renders the roadmap card in-app when it posts back, with status/last-attempt visible.

**Exit criteria:** upload → click → roadmap appears in the app. The manual copy-paste-to-Claude-Desktop loop is closed.

### Phase 3 — Persist the page index (`SessionPage`)
**Persist the same `PageIndexEntry`/`UrlRegistry` shapes already built and proven in P1** — this phase is migration + write path, not new modelling. Driven by durable drill-down, Teamwork rehydration, history/diff.

- **`SessionPage` table:** one row per crawled URL (`sessionId`, `path`, `hostId`/host, page metadata, `issueTypes JSON`). Index on `(sessionId, path)`. Bulk-insert via `createMany` (crawl-size-dependent — see §10). Persist the `UrlRegistry` (registry rows or a compact JSON column) so external/resource/redirect refs survive, not just internal pages.
- **Denormalized scalar columns on `Session`:** `siteHost`, `totalUrls`, `criticalCount`, `warningCount`, `noticeCount` (so history/list queries don't deserialize blobs). The composite score is dropped (D8), so it is *not* among these.
- Keep `Session.result` blob for the full export; `SessionPage` + registry become the queryable source of truth for drill-down/Teamwork/diff.
- **Historical tolerance:** no backfill of old sessions. Every consumer (dashboard drill-down, diff, Teamwork) must tolerate sessions lacking a persisted page index / compact refs and degrade gracefully.

**Exit criteria:** per-URL drill-down in the dashboard; complete affected-URL sets queryable for Teamwork without deserializing the blob.

### Phase 4 — Structured recommendations + Teamwork push
- **Structured `Recommendation` objects** replace the flat `string[]`: `{ issueType, severity, count, affectedUrlRefs[], effort, fixGuidance }` from the `ISSUE_RECOMMENDATIONS` map (move that map to `lib/constants/issue-recommendations.ts`). Backwards-compat string array kept as a display shim if needed.
- **Teamwork push (D7):** **one task per issue type**, affected URLs (rehydrated from the `UrlRegistry`) in the body. Where an issue has both a source and a target (e.g. broken internal links), the body includes **both "pages where found" and the "broken target."** For duplicate-group issues (dup titles/H1/meta), keep one task but include **grouped sub-sections** in the body. **Human-confirmed only** — a button per issue / per roadmap, never auto-push. **Idempotency anchored to `clientId + issueType + affectedSetHash`, where the hash is over the compact canonical refs (§5a), not display URLs.** Create in a defined client tasklist (needs a `teamworkTasklistId` mapping on `Client`, or a fixed "SEO Inbox" fallback — Kevin to confirm). Show task status (queued/sent/failed); never synchronous in the request path (borrow ADA `queue-manager` pattern if needed).
- **Diff-driven resolution (later within this phase):** a subsequent crawl's diff can mark a pushed task resolved (anchored by the hash). Prevents the "stale task graveyard."

### Phase 5 — Per-client history, trend, diff surfacing
- **Client domain normalization first** (Codex/consensus): consistent `siteHost` matching (`www` handling) — prerequisite for trend.
- **Per-client SEO view** (`/clients/[id]` section): list of sessions, severity-count trend over time (Recharts, already in stack), persistent-issue table, **auto-diff latest-vs-previous** (the diff service already exists; just thread `clientId` and surface it from the client view).
- Optional: re-audit cadence badge (`lastSeoAuditAt` on `Client`), and read-only SEO signal on quarter-grid chips (no bidirectional sync).

### Phase 6 — Keyword Research (separate, linked route)
Built last (D5). Supersedes `FUTURE-keyword-research-tool-design.md`.

- **`/keyword-research` route** with its own upload flow: SEMRush Organic Positions + Organic Pages + **Keyword Gap (Missing)** + optional SF **page-text `.txt`** export. **Note:** the current parse route skips `.txt` files — Phase 6 must explicitly add `.txt` handling (don't assume existing upload parse covers it), with a size cap (e.g. top-N pages by organic traffic) before storing.
- **New `SemrushKeywordGapParser`** (Missing export → `gap_keywords[]`: keyword, volume, difficulty, intent).
- **`KeywordResearchSession` model**, **linked** to a technical-audit `Session` via **`technicalSessionId String?` + `clientId`** (concrete shape; add a `runGroupId` only later if many-to-many grouping is needed).
- **`buildKeywordResearchExport()`** payload + a **`keyword-strategy-memo` Claude skill** (same mint-token/PATCH handoff as Phase 2) → strategy memo rendered back in app.
- Reuses upload/session/client/export primitives from the technical-audit side.

### Cross-cutting (slot in as areas open)
- **Typed `supplemental_data` hook (D3) defined in Phase 1's payload shape.** It lives on **`AggregatedResult` and is passed through into `TechnicalAuditExport`**; per-URL supplemental values attach to `PageIndexEntry` (so DataForSEO SERP/backlink data and Node-check outputs can hang off individual URLs). All fields optional; no calls made this cycle. Defining it now avoids a breaking migration to stored payloads later.
- **Parser-routing test** asserting known filenames route to the correct parser (guards the load-bearing ordering comments, e.g. PageSpeedOpportunities-before-PageSpeed, SEMRush-after-filename parsers).
- **Investigate `mergeParserData` array dedupe** (`value.filter(i => !existingList.includes(i))` on object arrays is likely a no-op → silent double-count when two files emit arrays). Confirm + fix with explicit per-key merge strategy.
- **Fire-and-forget triggers** (pillar today; roadmap/Teamwork next) get status visibility / the ADA queue pattern before more are added.
- Decompose `AggregatorService` (god object, ~921 lines, `unknown` casts) **only** as Phase 2/3/4 force each area open — no standalone churn.

## 8. Handoff loop (reused pattern)

Copy pillar-analysis verbatim, don't abstract: JWT mint scoped to one analysis/session + action, short TTL, Bearer auth on GET payload + PATCH write-back. Add a `status` field (pending/processing/complete/error) on `SeoRoadmap` so the UI shows progress instead of an empty card when the skill stalls. Same shape reused for the keyword memo in P6.

## 9. Testing

- Unit: title/H1 join; URL-compaction round-trip (compact ↔ rehydrate); structured `Recommendation` builder; severity-count derivation; parser-routing fixture test; `mergeParserData` dedupe fix.
- Integration: mint-token → GET payload → PATCH write-back happy path + token-expiry + error status.
- Teamwork push: idempotency by `affectedSetHash`; "showing N of M" disclosure; never auto-fires.
- Follow existing repo test conventions (Vitest-style alongside `priority.service.test.ts`).

## 10. Risks / watch-outs

- **Crawl size unknown** → `SessionPage` row counts and bulk-insert cost unknown (Codex verify-item). Confirm max typical SF crawl before P3; use `createMany` + index; cap if needed (the crawler already hard-caps 1000 pages).
- **`priority.service` mislabeled as truth** → always render as "Suggested." The authoritative roadmap is the skill's output.
- **Teamwork task graveyard** → human-confirmed, one-per-issue-type, hash-anchored, diff-resolved. Never auto-push.
- **Handoff skill failures invisible** → status field + last-attempt timestamp + retry that doesn't require re-upload.
- **SEMRush format drift** → add "expected columns not found" warnings to SEMRush parsers (P6).
- **Domain normalization** must precede trend work (P5) or history fragments on `www`.
- **Subdomain conflation:** registry treats subdomains as distinct hosts by default; if Kevin wants them grouped as one site, that changes host matching and trend grouping.
- **Payload size:** complete URL sets can still be large; payload uses summary + complete refs and the skill must handle long ref lists. Watch for very large crawls.
- **Resource/external URLs** must resolve via `hostId`/`originalUrl` — path-only-under-session-host is invalid for CDN assets and external links.

# ER SEO Tools тАФ Claude Context

Internal SEO toolkit for Enrollment Resources. Next.js 15 App Router, TypeScript, Tailwind CSS (class-based dark mode), Prisma + SQLite. Deployed on RunCloud (not Vercel/serverless).

## Stack constraints
- **Node 22** on production
- **SQLite only** тАФ no Postgres/MySQL
- **No serverless** тАФ RunCloud + PM2
- **Google Chrome** must be installed on the server (`/usr/bin/google-chrome`) for ADA audits
- **Do not change the core stack** unless explicitly asked

## Architecture patterns
- **ADA audit polling:** POST creates record + enqueues a durable `ada-audit` job (dedup/group key `ada-audit:<id>`; enqueue failure flips the row to error + 500) тЖТ the job handler (`lib/jobs/handlers/ada-audit.ts`) updates `progress`/`progressMessage` as it runs (every write fenced by `status='running'` тАФ first terminal writer wins) тЖТ client polls `/api/ada-audit/[id]` every second тЖТ `AuditPoller` renders live progress bar with elapsed + estimated time. Standalone concurrency = `ADA_AUDIT_CONCURRENCY` (default 2). Orphaned standalone audits/PDF rows (zero active jobs in their `ada-audit:<id>` group, >5 min old тАФ AdaAudit has no `updatedAt`; job state is the liveness source) are failed by `lib/ada-audit/standalone-recovery.ts`, called from `resetStaleAudits()` + `recoverQueue()`
- **Site audit queue:** Only one site audit runs at a time, enforced at the DB level by the `site-audit-discover` job's conditional claim (`NOT EXISTS` over transient statuses). `enqueueAudit()` creates the record in `queued` status тЖТ `processNext()` is a stateless promoter (no mutex): when nothing is transient it enqueues a `site-audit-discover` job (dedupKey `discover:<id>`) for the oldest queued audit; `finalizeSiteAudit` kicks it on completion. `SiteAuditPoller` shows queue position + active audit progress. `SiteAuditForm` polls `/api/site-audit/queue` every 5s to show a live banner. `SiteAuditHistory` smart-polls every 8s when active audits exist.
- **Site-audit phase model:** `queued тЖТ running тЖТ (pdfs-running | lighthouse-running, in that priority) тЖТ complete`. The `site-audit-discover` job claims `queuedтЖТrunning`, discovers pages, persists `discoveredUrls`+`pagesTotal` (the finalizer's discovery guard: a `running` audit with null `discoveredUrls` is never finalized), creates child rows, and fans out one `site-audit-page` job per URL. Each page job runs axe, dispatches PDFs FIRST (pdfsTotal-before-pagesComplete invariant), settles child + counters in one array-form transaction, then enqueues PSI (`lib/ada-audit/lighthouse-queue.ts`). `finalizeSiteAudit` is the single decision point: it requires `pagesDone && pdfsDone && lighthouseDone` to flip to `complete`, otherwise picks `pdfs-running` (priority) or `lighthouse-running` as the transient status; it reads scalars only and loads children once, at completion. PSI concurrency is `PSI_CONCURRENCY` (default 6); page concurrency is `SITE_AUDIT_CONCURRENCY` (default 1, prod 2). **C11 PR1 `seoOnly` render-only mode:** a `SiteAudit.seoOnly` audit (set by `POST /api/site-audit {seoOnly:true}` тАФ the Site Audit form's Scan Type = SEO (C16: `SeoScanForm` retired) тАФ or a seoOnly schedule (C11 PR 2a); `seoOnlyтЗТseoIntent` forced at `queueSiteAuditRequest`) takes a render-only page path (`runAxeAudit({renderOnly})`тЖТ`kind:'rendered'`: nav+settle+harvest only; NO axe/screenshots/PDF-dispatch/PSI), so pages settle straight to `complete` with `pdfsTotal===0 && lighthouseTotal===0`. The page job reads `seoOnly` off the PARENT row (authoritative, not the payload); the finalizer writes a null `summary` and **skips ALL ADA output тАФ NO `tool:'ada-audit'` CrawlRun, no carry-forward** (ADA report/csv/vpat gate on the ada run existing тЖТ they 409 `no_findings_run` for seoOnly), but still fires `enqueueBrokenLinkVerify` (the live-scan builder тАФ the ONLY output; result at `/seo-audits/results/run/[liveScanRunId]`, readiness signalled by `liveScanRunId` on `GET /api/site-audit/[id]`). **C16:** `/ada-audit/site/[id]` OWNS seoOnly routing тАФ transient тЖТ `SiteAuditPoller`; complete тЖТ server redirect to the run results page (or `SeoPhaseBanner` while the verifier builds), branch placed BEFORE the ADA summary resolution (`seo-only-view.ts`); all producers link the site page (only surfaces with the run id in hand тАФ unified recents, `ScheduledScansCard` тАФ link the run page directly); unified recents badges seoOnly rows тАЬSite SEOтАЭ.
- **Scheduled scans (C2):** client scan schedules are plain `Schedule` rows (`name: null`, `clientId` set, `jobType: 'scheduled-site-audit'`, payload `{clientId, domain, wcagLevel}`) managed via `GET/POST /api/clients/[id]/schedules` + `PATCH/DELETE тАж/[scheduleId]` (cookie-gated; cadence must be literal `weekly:`/`monthly:` тАФ `daily@`/`every:*` are 400 `cadence_not_allowed` until C3 makes blobs prunable; one schedule per (client,domain) is best-effort app-level; resume recomputes `nextRunAt` and 409s on archived clients; DELETE cancels the `schedule:<id>` job group and SetNulls historical audits to manual-class). The tick fires the wrapper job тЖТ `queueSiteAuditRequest()` тЖТ SiteAudit born with `scheduleId` + `requestedBy: 'scheduled'`; a duplicate in-flight audit consumes the slot (no catch-up). `monthly:<1-28>@HH:MM` cadence + `cadenceClass()` live in `lib/jobs/scheduler.ts`. - **Stale audit recovery:** `updatedAt` auto-updates on every Prisma write (raw-SQL counter bumps set it manually) тАФ that's the heartbeat. Both `resetStaleAudits()` (every 10 min via the `stale-audit-reset` scheduled job, 5-min threshold) and `recoverQueue()` (once at startup) apply ONE generic treatment to all transient parents (`running` included since Phase 3): outstanding durable jobs in `site-audit:<id>` тЖТ resume; zero jobs тЖТ one finalize attempt (drained-but-unfinalized completes); still transient тЖТ `failSiteAudit()`, the shared destructive path (conditional parent flip that never clobbers terminal rows, then `failOrphanAdaAudits` + `failOrphanPdfAudits` + `cancelJobsByGroup` + batch close + promoter kick). A failed job count never destroys a parent тАФ skip and retry next pass.
- **Durable job queue:** `Job` + `Schedule` tables (`lib/jobs/`) with a single in-process worker тАФ conditional-update claim, attempt-fenced heartbeat/settle, per-type concurrency/timeout/backoff, `onExhausted` domain hooks, and a 60s schedule tick (exactly-once-per-slot via `@@unique([scheduleId, scheduledFor])`). ## ADA Audit specifics
- axe-core runs inside headless Chrome (puppeteer-core). Navigation uses `waitUntil: 'domcontentloaded'` (30 s budget) followed by a best-effort `waitForNetworkIdle({ idleTime: 500, timeout: 5_000 })` settle via `postLoadSettle` in `lib/ada-audit/page-load.ts`. The settle is non-fatal тАФ analytics/chat poll traffic on real client sites would otherwise prevent network-idle from ever firing. CSS and fonts that block first-paint are already in the DOM at DCL; color-contrast checks still work.
- `wcagLevel`: `wcag21aa` (default, "Required") runs `['wcag2a','wcag2aa','wcag21a','wcag21aa']`; `wcag22aa` ("Aspirational") adds `['wcag22aa','best-practice']`
- Progress written to DB at each phase (5тЖТ10тЖТ20тЖТ75тЖТ82тЖТ95тЖТ100%); client polls every second and shows live bar + estimated time
- `runnerType` column on `AdaAudit`/`SiteAudit`: `'browser'` for new audits, `'jsdom'` for legacy
- `domElementCount` stored in result JSON тАФ values < 50 trigger an "unreliable result" warning (JS-rendered SPA)
- Score formula: v1 (`scoring.ts`, frozen) = weighted penalty per impact ├╖ log10(totalElements); **live scoring is v2-formula/v3-version** (`scoring-v2.ts`, see Key files) тАФ density-based, computed at the findings-mapper seam (`ada-mapper.ts`), persisted on `CrawlRun.score`/`scoreBreakdown`

## Do not
- Use interactive `prisma.$transaction(async tx => ...)` anywhere тАФ array-form `$transaction([...])` only. Interactive transactions hold SQLite's write lock across event-loop round-trips; concurrent pdfjs parsing starves the loop and every other writer times out ("Operations timed out", 2026-06-10 production incident). Express conditional logic in SQL (`EXISTS` predicates) instead, and set `updatedAt` manually in raw statements (`Date.now()` тАФ storage is integer ms; raw SQL bypasses `@updatedAt`).
- Add AI/LLM API features (Anthropic or any other provider) тАФ decided 2026-07-08: no plans to use any AI API; all AI stays the skill-handoff clipboard flow (pat_/srt_/krt_/qct_). Reopening this is a Kevin decision recorded in the roadmap tracker's Gated decisions, never a session call
- Use `npm ci` on production (RunCloud uses `npm install`)
- Trust request origin headers for share URLs тАФ use `NEXT_PUBLIC_APP_URL`
- Increase `BROWSER_POOL_SIZE` above 4 without first checking VPS memory headroom тАФ each Chrome page is ~150-200 MB resident
